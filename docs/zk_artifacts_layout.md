# ZK Artifacts Directory Structure (ZK-041)

This document defines the standardized directory structure for compiled circuits, proving metadata, and generated assets consumed by the PrivacyLayer SDK.

## Overview

All ZK artifacts are organized under `artifacts/zk/` with versioned subdirectories to support circuit evolution while maintaining backward compatibility.

## Directory Structure

```
artifacts/
└── zk/
    └── v{version}/                    # Versioned artifact directory (e.g., v1)
        ├── circuits/                  # Compiled circuit artifacts
        │   ├── commitment/
        │   │   └── commitment.json    # Compiled circuit (ACIR + ABI)
        │   ├── withdraw/
        │   │   └── withdraw.json      # Compiled circuit (ACIR + ABI)
        │   └── merkle/
        │       └── merkle.json        # Compiled circuit (ACIR + ABI)
        ├── manifests/                 # Circuit metadata and checksums
        │   └── manifest.json          # Global manifest for all circuits
        ├── fixtures/                  # Test vectors and golden inputs
        │   ├── commitment/
        │   │   └── test_vectors.json  # Test vectors for commitment circuit
        │   ├── withdraw/
        │   │   └── test_vectors.json  # Test vectors for withdrawal circuit
        │   └── merkle/
        │       └── test_vectors.json  # Test vectors for merkle circuit
        └── proving_keys/             # Proving and verification keys
            ├── commitment/
            │   ├── vk                 # Verification key
            │   └── pk                 # Proving key
            ├── withdraw/
            │   ├── vk                 # Verification key
            │   └── pk                 # Proving key
            └── merkle/
                ├── vk                 # Verification key
                └── pk                 # Proving key
```

## Versioning

Artifact versions are tracked via the `version` field in `sdk/src/artifacts.ts`. When circuit definitions change incompatibly:

1. Increment `ZK_ARTIFACT_VERSION` in `sdk/src/artifacts.ts`
2. Create a new versioned directory (e.g., `artifacts/zk/v2/`)
3. Build scripts automatically use the new version
4. SDK can still access older versions by specifying the version parameter

## File Descriptions

### Circuit Artifacts (`circuits/`)

- **`{circuit_name}.json`**: Compiled Noir circuit containing ACIR (Arithmetic Circuit Intermediate Representation) and ABI (Application Binary Interface). Generated by `nargo compile`.

### Manifests (`manifests/`)

- **`manifest.json`**: Global manifest containing:
  - Artifact version
  - Backend type (e.g., barretenberg)
  - Circuit metadata (paths, checksums, configuration)
  - Root depth for Merkle trees

### Fixtures (`fixtures/`)

- **`test_vectors.json`**: Golden test vectors and inputs for circuit testing. Used by SDK tests to verify circuit behavior.

### Proving Keys (`proving_keys/`)

- **`vk`**: Verification key for Groth16 proofs. Used for proof verification.
- **`pk`**: Proving key for Groth16 proofs. Used for proof generation.

## SDK Integration

The SDK uses the centralized path configuration in `sdk/src/artifacts.ts`:

```typescript
import { 
  getKnownCircuitPath, 
  CIRCUIT_NAMES,
  ZK_ARTIFACT_VERSION 
} from './artifacts';

// Get current version withdraw circuit
const withdrawPath = getKnownCircuitPath(CIRCUIT_NAMES.WITHDRAW);

// Get specific version
const withdrawPathV1 = getKnownCircuitPath(CIRCUIT_NAMES.WITHDRAW, '1');
```

## Build Script Integration

Build scripts (`scripts/rebuild-zk.sh`, `scripts/build-circuits.sh`) use the layout defined in `sdk/src/artifacts.ts`:

1. Compile circuits using `nargo compile`
2. Copy compiled circuits to `artifacts/zk/v{version}/circuits/{circuit_name}/`
3. Generate manifest in `artifacts/zk/v{version}/manifests/`
4. (Optional) Generate proving keys in `artifacts/zk/v{version}/proving_keys/{circuit_name}/`

## Migration from Old Layout

The previous flat layout (`artifacts/zk/withdraw.json`) is deprecated. To migrate:

1. Create new versioned directory structure
2. Move existing artifacts to new locations
3. Update build scripts to output to new locations
4. Update SDK imports to use centralized path functions
5. Remove old flat artifacts after validation

## Benefits

- **Centralized Configuration**: Single source of truth for artifact paths
- **Version Support**: Easy to support multiple circuit versions simultaneously
- **Testability**: Path resolution can be tested independently
- **Maintainability**: Layout changes are centralized in one place
- **Backward Compatibility**: Can maintain multiple versions during transitions
